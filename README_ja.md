# Stock Strategy Analyzer

> 日本株投資家として自分が直面した課題を解決するために開発したFinTechツール。東証全銘柄（3,800銘柄以上）を8つの定量的投資戦略でスコアリングし、「勘」ではなく「データ」で投資判断を下せる環境を構築しました。

## 🎯 なぜ作ったか（課題定義）

日本株の個人投資家として、特定の銘柄が特定の投資戦略に合致するかを判断するには、複数のテクニカル指標を手動で確認する必要がありました。このプロセスは**時間がかかる・一貫性がない・数千銘柄には到底スケールしない**という問題を抱えていました。

**課題を一文で言うと：** 東証全銘柄を複数の投資戦略でスコアリングし、「エントリーシグナルが近い銘柄」を自動で教えてくれるツールが存在しなかった。

既存のスクリーナー（TradingView、株探等）はフィルタリング機能を持つが、**カスタム戦略への適合度スコアリング**と**シグナル接近タイミングの検出**を組み合わせたツールは日本株市場向けには存在しなかった。

そこで要件を自分で定義し、アーキテクチャを設計し、AI協働開発で構築しました。

## ✅ 何ができるか

| 機能 | 内容 |
|---|---|
| **多戦略スコアリング** | 東証全銘柄（3,800銘柄以上）を8つの定量的投資戦略（買い4・空売り4）でスコアリング |
| **適合度スコア算出** | 複数のテクニカル指標を統合し、戦略ごとに0〜100%の適合度スコアを算出 |
| **シグナル接近検出** | 直近1〜3ヶ月のデータから、エントリーシグナルが近い銘柄を自動で検出・推定到達日数を表示 |
| **自動日次スキャン** | 平日22:00にバッチ処理が自動実行。翌朝には結果が揃っている状態を実現 |
| **Web UI** | 戦略別適合度ランキング・銘柄別詳細分析・シグナル接近銘柄一覧 |
| **CLI** | 単一銘柄の即時分析・戦略別フィルタリング |

## 🏗️ 主な設計判断とその理由

プロダクトとして重要な意思決定を記録します。

**なぜバイナリーシグナルではなく適合度スコア（0〜100%）にしたのか？**
「買い/売り」の二値判定は偽の精度を生む。スコア形式にすることで、85%の銘柄を60%の銘柄より優先できる。ポジションサイジングの判断材料にもなる。「もうすぐ準備完了」という状態も可視化できる。

**なぜ買いと空売りの両方（計8戦略）を網羅したのか？**
相場環境によって有効な戦略が異なる。トレンド相場ではブレイクアウト・モメンタム戦略が機能し、レンジ相場では逆張り戦略が優位になる。両方向をカバーすることで、相場環境に関わらず機会を検出できる設計にした。

**なぜベクトル化バックテストエンジンを採用したのか？**
3,800銘柄以上をスキャンするには処理速度が制約になる。ベクトル化エンジンにより、全銘柄スキャンを**1.5〜2時間以内**に完了できる。これは「夜間に自動実行して翌朝に使える状態にする」という要件を満たすための設計判断であり、オプションではなく必須要件だった。

**なぜデータソースのフォールバック機能（Stooq → yfinance）を設けたのか？**
日本株の無料APIはデータ取得の安定性が保証されない。単一ソースに依存するとバッチ処理が途中で失敗する。フォールバック構成により、日次バッチが手動介入なしに完走する信頼性を確保した。

**なぜ手動実行ではなく自動スケジューリングにしたのか？**
「実行しなければならない」ツールは、いずれ使われなくなる。平日22:00の自動実行により、翌朝に結果が揃っている状態をデフォルトにした。ユーザー（自分）の行動コストをゼロにすることが継続使用の条件だった。

## 📊 実際の活用状況

- 東証全銘柄（3,800銘柄以上）を毎日自動スキャン（処理時間：1.5〜2時間）
- 毎朝、シグナル接近リストで「監視対象」を特定するワークフローに組み込み
- 適合度スコアを活用し、低適合の銘柄を事前に除外。高確信度のトレードに集中
- ファンダメンタル分析の前段階として、定量スクリーニングのメインツールとして運用中

## 🧠 プロダクトオーナーとして学んだこと

**スコープの規律が品質を決める。** 当初は12の戦略を設計した。実際に使ってみて8つに絞った。複雑性の増加がシグナルの質向上につながらないと判断したため。多い戦略より、精度の高い戦略を優先した。

**パフォーマンスはプロダクト要件である。** 「夜間に完走する」という制約を設計の最初期に定めたことで、ベクトル化エンジンの採用が必然になった。性能要件を後から考えるのではなく、要件定義に含めることの重要性を実感した。

**実際の使用が仕様の盲点を教えてくれる。** 上場廃止銘柄、データ欠損、極端な外れ値など、実際に自分のワークフローで使い始めて初めて見えたエッジケースが多数あった。ユーザー（自分）として使い続けることが最良のフィードバックループだった。

**AI協働開発は「何を作るか」の判断を変えない。** Claude Codeを使ったことで実装コストが大幅に下がったが、「何を作るか・なぜ作るか・どう設計するか」の意思決定はすべて自分が担った。AIは実装パートナーであり、プロダクトの方向性を決めるのは人間の役割だと確信した。

## ⚙️ セットアップと使い方

<details>
<summary>クリックして展開</summary>

### 依存パッケージのインストール

```bash
pip install -r requirements.txt
```

### 銘柄リストの配置

[JPX](https://www.jpx.co.jp/markets/statistics-equities/misc/01.html) から `data_j.xls` をダウンロードし、プロジェクトルートに配置してください。

### Web UIの起動

```bash
python -m src.web.app
# ブラウザで http://localhost:5000 を開く
```

### CLIの使用

```bash
# 銘柄を分析（例：NTT 9432）
python -m src.ui.cli analyze 9432

# 戦略でフィルタリング
python -m src.ui.cli filter-stocks 新高値ブレイク --threshold 70 --top 20
```

### バッチ処理の手動実行

```bash
python -m src.batch.daily_batch
```
